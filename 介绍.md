# LongVideoEvalKit 简介

## 1. 定位与适用范围

`LongVideoEvalKit` 是一个面向长视频生成实验的特征驱动评测工具包，目标是为本地测试、消融实验和回归跟踪提供稳定、可复现的分析流程。

目前工具包包含两条评测路径：

1. `longvideo-eval run`
   面向长视频生成集合，结合 prompt、抽帧、分段特征和长时一致性指标做模型级评测。
2. `longvideo-eval paired-run`
   面向成对视频评测，输入一个 GT 目录和一个预测目录，输出 `PSNR / SSIM / LPIPS / FVD` 等成对指标。

这两条路径的关注点不同：

- `run` 更适合“长视频模型整体表现分析”
- `paired-run` 更适合“预测结果与参考视频有多接近”

需要先区分三类指标：

1. 工具包原生代理指标：`quality_proxy.*`、`colorhist_*`、`clip_*`、`dinov2_*`、`drift_*`、`repetition_*`、`efficiency.*`。
2. 元数据与记录字段：`metadata.*`、`model`、`prompt_id`、`video_path`、`status`。
3. 官方 VBench 指标：工具包内部并未重实现，而是在启用时调用官方 VBench，保留原始输出，并将标准化后的 `vbench.*` 字段合并进报告。

这个区分非常重要：

- 工具包原生指标适合做相对比较、问题定位、消融分析和失败诊断。
- 工具包原生指标不应直接当作官方 VBench 分数来报告。
- 如果某个分数来自官方 VBench，报告时应明确标注为 VBench。

另外，`paired-run` 的输出不属于上面这套 prompt-aware 长视频指标体系，而是单独的一组配对视频指标：

- `psnr_mean`
- `ssim_mean`
- `lpips_mean`
- `fvd_pair`
- `fvd_dataset`

因此在阅读报告时，要先确认自己看到的是：

- 长视频集合评测结果
还是
- 成对视频评测结果

不能把这两类数值混在一起解释。

## 1.1 `paired-run` 的定位

`paired-run` 适用于以下场景：

- 视频预测
- 视频重建
- 有 GT 的条件生成
- 模型 A 输出与模型 B 输出的成对比较

它会自动执行三层配对：

1. 完整文件名匹配
2. 去掉扩展名后的 stem 精确匹配
3. 去掉扩展名后的 stem 从后向前后缀匹配

并输出配对报告：

- `pairing_report.json`
- `pairing_report.csv`

其中会明确列出：

- 成功但非精确文件名匹配的样本
- 未匹配上的 GT 视频
- 未匹配上的预测视频

如果配对视频时长不一致：

- 工具包会取较短时长
- 将长视频从开始截取到同样时长后再计算指标

## 2. 评测流程

对每个视频，工具包采用如下统一流程：

1. 读取视频元信息，并按 `sample_fps` 抽帧。
2. 按 `segment_seconds` 将抽样时间戳切分成等长时间段。
3. 每个时间段最多保留 `max_frames_per_segment` 帧。
4. 提取帧级特征。
5. 对每个时间段内的帧特征求均值。
6. 对时间段特征做 L2 归一化。
7. 基于时间段级特征计算长时一致性、漂移、重复、文本对齐和质量诊断指标。

形式化地说，若第 `k` 个 segment 的帧索引集合为 `S_k`，第 `i` 帧特征为 `f_i`，则该段特征定义为：

```text
z_k = normalize(mean_{i in S_k}(f_i))
```

很多指标都是相对于第一个 segment 的特征 `z_1` 来定义的，它相当于长时比较时的初始锚点。

## 3. 命名规则

### 3.1 单视频指标

- `*.mean`：对所有时间段，或所有合法 pair，取平均。
- `*.end`：最后一个时间段的值。
- `*.drop`：首段值减去末段值。
- `drift_*`：与首段相似度的补数，即 `1 - similarity`。
- `repetition_*`：对时间上相隔较远的 segment pair 的相似度统计。
- `quality_proxy.head`：首段质量。
- `quality_proxy.tail`：末段质量。

### 3.2 模型汇总指标

`model_summary.csv` 的生成方式，是对同一 `model` 下所有成功视频的每个数值列再取一次平均。

因此：

- `clip_f.mean.mean` 的含义是“先对每个视频算 `clip_f.mean`，再对这些视频均值继续求平均”。
- `quality_proxy.tail.mean` 的含义是“该模型所有视频的尾段质量平均值”。

## 4. 各指标的专业解释

### 4.1 元数据字段

这些字段是描述信息，不是质量分数：

- `metadata.duration_sec`：视频总时长，单位秒。
- `metadata.video_fps`：视频原始帧率。
- `metadata.num_frames`：视频原始总帧数。
- `metadata.width`、`metadata.height`：空间分辨率。

它们主要用于做公平比较和效率归一化分析。

### 4.2 `quality_proxy.*`

这是工具包内部实现的轻量级无参考质量代理分数，不是 VBench Imaging Quality，也不是 MUSIQ。

对每一帧：

```text
gray = grayscale(frame)
sharpness = Var(Laplacian(gray))
sharp_score = sharpness / (sharpness + 100)

mean_luma = mean(gray) / 255
exposure_score = 1 - min(abs(mean_luma - 0.5) / 0.5, 1)

frame_quality = 0.7 * sharp_score + 0.3 * exposure_score
```

对每个 segment，其质量分数定义为该段内所有帧质量的平均值。

输出字段包括：

- `quality_proxy.mean`：整段视频的平均 segment 质量。
- `quality_proxy.head`：首段质量。
- `quality_proxy.tail`：末段质量。
- `quality_proxy.segment_std`：各段质量随时间变化的标准差。
- `quality_proxy.delta_abs = |head - tail|`。
- `quality_proxy.delta_drop = head - tail`。

解释方式：

- `quality_proxy.mean` 越高，通常代表画面越清晰、曝光越平衡。
- `quality_proxy.segment_std` 越低，通常代表视频整体质量越稳定。
- `quality_proxy.delta_drop > 0` 表示尾段质量比首段差。
- `quality_proxy.delta_drop < 0` 表示尾段质量反而优于首段。

局限性：

- 它并不直接衡量语义正确性、运动真实性、时间连续性或人类偏好。
- 它可能高估“很锐利但语义错误”的视频。
- 它也可能低估刻意偏暗、低照度或风格化场景。

### 4.3 `colorhist_lc.*`

这组指标基于 HSV 颜色直方图，用于衡量长时外观一致性。

对每一帧：

```text
feature = normalize(HSV_histogram(frame, bins=(8, 8, 8)))
```

对每个 segment：

```text
z_k = normalize(mean frame histogram in segment k)
```

定义与首段的相似度：

```text
s_k = cos(z_k, z_1)
```

输出字段包括：

- `colorhist_lc.mean = mean_k(s_k)`
- `colorhist_lc.end = s_last`
- `colorhist_lc.drop = s_1 - s_last`

解释方式：

- 值越高，说明视频整体颜色分布越接近开头。
- 这类指标适合检测曝光漂移、色调变化、背景外观大变等问题。
- 但它不是语义指标。两个语义完全不同的画面，也可能拥有相似的颜色直方图。

### 4.4 `drift_colorhist.*`

这组指标将颜色相似度转成“漂移量”：

```text
drift_k = 1 - cos(z_k, z_1), for k >= 2
```

输出字段包括：

- `drift_colorhist.mean`
- `drift_colorhist.end`
- `drift_colorhist.max`

解释方式：

- 越低越好。
- `drift_colorhist.end` 高，说明尾段颜色统计与首段差异大。
- `drift_colorhist.max` 高，说明中间至少出现过一次明显外观漂移。

### 4.5 `repetition_colorhist.*`

这组指标用于检测时间上相隔较远的段之间是否发生重复。

若两个 segment 至少相隔 `repetition_min_gap_segments`，则将其相似度纳入统计：

```text
pair_sim(i, j) = cos(z_i, z_j), with j >= i + min_gap
```

阈值记为 `repetition_threshold`。

输出字段包括：

- `repetition_colorhist.ratio`：合法远距离 pair 中，相似度超过阈值的比例。
- `repetition_colorhist.mean_sim`：所有合法远距离 pair 的平均相似度。
- `repetition_colorhist.max_sim`：所有合法远距离 pair 的最大相似度。
- `repetition_colorhist.num_pairs`：合法 pair 数量。

解释方式：

- 重复度高，可能意味着循环播放、背景冻结、周期性失败或模式坍塌。
- 但基于颜色的重复检测更偏向“外观相似”，不一定能判断语义是否真的重复。

### 4.6 `clip_f.*`

这组指标基于 CLIP 图像特征，用来衡量长时语义一致性。

每个抽样帧都会送入 CLIP 图像编码器，得到帧级 embedding；然后按 segment 求均值并归一化。

仍然定义与首段的相似度为：

```text
s_k = cos(z_k, z_1)
```

输出字段包括：

- `clip_f.mean`
- `clip_f.end`
- `clip_f.drop`

解释方式：

- `clip_f.mean` 高，表示视频整体语义在平均意义上更接近开头。
- `clip_f.end` 高，表示结尾仍与开头在语义上保持相似。
- `clip_f.drop` 高，表示随着时间推进，语义一致性有明显衰减。

与 `colorhist_lc` 相比，`clip_f` 通常更能反映人物身份、场景语义和主要内容的漂移。

局限性：

- CLIP 并不是专门为视频时间建模设计的。
- 它可能对微小几何变化、局部身份细节变化不够敏感。
- 它也可能更偏向捕捉粗粒度语义，而不是严格的像素级保真。

### 4.7 `drift_clip.*`

这组指标将 CLIP 相似度转成语义漂移量：

```text
drift_k = 1 - cos(z_k, z_1), for k >= 2
```

输出字段包括：

- `drift_clip.mean`
- `drift_clip.end`
- `drift_clip.max`

解释方式：

- 越低越好。
- 这组指标通常是分析长视频语义漂移时最实用的一组代理指标。
- `drift_clip.max` 特别适合捕捉“均值看着还行，但中间某一段已经灾难性跑偏”的情况。

### 4.8 `repetition_clip.*`

这组指标使用 CLIP segment 特征，对远距离时间段之间的语义重复进行检测。

输出字段包括：

- `repetition_clip.ratio`
- `repetition_clip.mean_sim`
- `repetition_clip.max_sim`
- `repetition_clip.num_pairs`

解释方式：

- 高重复往往意味着循环、语义回摆、状态复用或运动停滞。
- 若 `repetition_clip` 很高，同时 `clip_f` 也很高，模型可能过于静止，几乎不发生有效变化。
- 若 `repetition_clip` 很高，但 `clip_f` 较低，则模型可能是在少数几个语义状态之间来回摆动，而不是自然推进。

### 4.9 `clip_t.*`

这组指标使用 CLIP 图文相似度来衡量视频对 prompt 的时间一致性。

设 prompt 的 CLIP 文本特征为 `t`，第 `k` 段图像特征为 `z_k`，则：

```text
a_k = cos(z_k, t)
```

输出字段包括：

- `clip_t.mean`
- `clip_t.end`
- `clip_t.drop`

解释方式：

- `clip_t.mean` 越高，说明整段视频平均更贴合文本提示。
- `clip_t.end` 越高，说明视频结尾仍然保持 prompt 对齐。
- `clip_t.drop` 越大，说明模型前期还能贴 prompt，但后期逐渐偏离。

局限性：

- CLIP 图文相似度只是 prompt faithfulness 的代理信号。
- 对细粒度动作顺序、数量关系、精确空间关系和复杂长程事件，它并不总是可靠。

### 4.10 `dinov2_lc.*`、`drift_dinov2.*`、`repetition_dinov2.*`

这组指标与 CLIP 图像指标形式相同，但底层特征来自 DINOv2，而非 CLIP。

解释方式：

- `dinov2_lc.*` 往往更强调视觉结构和外观稳定性。
- `drift_dinov2.*` 更偏向反映非文本条件下的视觉漂移。
- 将 DINOv2 与 CLIP 指标对照，常常能帮助区分“纯视觉外观漂移”和“与 prompt 相关的语义漂移”。

这组指标也属于工具包原生指标，不是 VBench 指标。

### 4.11 `paper.*`

这组字段是后续新增的“精确版论文指标”，不是早期的轻量代理分数。

它们更贴近 Context Forcing、Rolling Forcing、Deep Forcing、Relax Forcing 等论文里的写法，而不是只做快速 smoke test。

#### 4.11.1 `paper.aesthetic_quality_drift_abs`、`paper.imaging_quality_drift_abs`

这两项对应“首尾质量差”的精确实现。

对于任意质量评分器 `Q`，统一定义为：

```text
DeltaQ(v) = |Q(v_first_5s) - Q(v_last_5s)|
```

工具包目前给出两个具体版本：

- `paper.aesthetic_quality_drift_abs`：对首 5 秒与末 5 秒 clip 分别跑官方 VBench `aesthetic_quality` 后取绝对差。
- `paper.imaging_quality_drift_abs`：对首 5 秒与末 5 秒 clip 分别跑官方 VBench `imaging_quality` 后取绝对差。

解释方式：

- 越低越好。
- 值越大，说明视频结尾与开头在官方质量评分器下差异越明显。
- 相比 `quality_proxy.delta_abs`，这两项更贴近论文口径，因为它们使用的是官方质量打分器，而不是轻量级启发式代理。

注意：

- 这组字段虽然复用了官方 VBench 的 clip 级打分，但它们本身仍属于工具包派生指标，不是原生整视频 `vbench.*` 字段。

#### 4.11.2 `paper.imaging_quality_drift_std`、`paper.imaging_quality_num_clips`

这项不是只看开头和结尾，而是看整段视频过程中画质是否忽高忽低。

视频会被切成连续 clips，对每一段计算官方 `imaging_quality`：

```text
q_i = IQ(c_i)
paper.imaging_quality_drift_std = Std(q_1, ..., q_n)
```

解释方式：

- 越低越好。
- 值高通常意味着中间过程存在间歇性模糊、伪影爆发、曝光波动等问题。
- `paper.imaging_quality_num_clips` 记录实际参与统计的 clip 数量。

#### 4.11.3 `paper.drift_clip_first_last`

这是“CLIP 首尾语义漂移”的精确实现：

```text
Drift(v) = 1 - cos(phi_CLIP(v_first_5s), phi_CLIP(v_last_5s))
```

其中每个 clip 的特征由 clip 内帧级 CLIP image embedding 平均后再归一化得到。

解释方式：

- 越低越好。
- 与 `drift_clip.end` 相比，这项更严格，因为它直接比较明确的首 clip 和末 clip，而不是把所有后续 segment 相对首段做统计。

#### 4.11.4 `paper.repetition_clip_global`、`paper.repetition_clip_global.num_pairs`

这是按照论文风格实现的全局 CLIP 重复度：

```text
Rep(v) = Avg_{i<j} cos(phi_CLIP(c_i), phi_CLIP(c_j))
```

解释方式：

- 越高表示视频越容易回到相似的语义状态。
- 值过高时，常见现象是循环、冻结、状态复用或语义推进不足。
- `paper.repetition_clip_global.num_pairs` 记录参与平均的 clip pair 数量。

与 `repetition_clip.*` 相比，这一版更接近论文中的 all-pairs 定义，而且不依赖阈值。

#### 4.11.5 `paper.balance.mean`

这项实现的是 Relax Forcing 风格的平衡指标：

```text
Balance = minmax(Drift) + minmax(Repetition)
```

解释方式：

- 越低越好。
- 漂移很低但重复很高，并不是真正意义上的好结果。
- 重复不高但漂移很大，也同样不是理想结果。
- 这项指标就是为了量化这两类失败模式之间的权衡。

注意：

- `paper.balance.mean` 只在 `model_summary.csv` 层计算，不是单视频原子指标。
- 由于使用了 min-max 归一化，它只在同一批模型一起比较时才有意义。

#### 4.11.6 `paper.dinov2_cf.*`

这组字段实现的是 Context Forcing 风格的 DINOv2 指标，采用窗口式采样。

对任意评估时刻 `t`，取一个中心窗口 `[t-0.5s, t+0.5s]`。记首帧为 `V_0`，窗口内帧为 `V_tau`，则：

```text
DINOv2(t) = Avg_{tau in [t-0.5s, t+0.5s]} cos(phi_DINO(V_tau), phi_DINO(V_0))
```

工具包输出：

- `paper.dinov2_cf.mean`
- `paper.dinov2_cf.end`
- `paper.dinov2_cf.drop`

解释方式：

- 值越高，说明按 Context Forcing 的评估协议，视频在结构与外观上越接近开头。
- 与 `dinov2_lc.*` 相比，这一版更接近论文原始定义，因为它使用了“以时间点为中心的局部窗口”，而不是只用 segment mean feature。

#### 4.11.7 与旧版工具包代理指标的对照

精确版 paper 指标并不意味着旧版工具包指标就没用了。两者更适合解决不同问题。

| 精确版 paper 指标 | 旧版工具包指标 | 主要相似点 | 主要不同点 | 推荐用法 |
|---|---|---|---|---|
| `paper.aesthetic_quality_drift_abs` / `paper.imaging_quality_drift_abs` | `quality_proxy.delta_abs`, `quality_proxy.delta_drop` | 都在看“首尾质量变化”。 | `paper.*` 用官方 VBench 质量打分器给 clips 打分；`quality_proxy.*` 用的是轻量级锐度/曝光启发式代理。 | 做 benchmark、论文表格时优先 `paper.*`；快速调试时保留 `quality_proxy.*`。 |
| `paper.imaging_quality_drift_std` | `quality_proxy.segment_std` | 都在看“质量是否随时间波动”。 | `paper.*` 基于官方 `imaging_quality`；`quality_proxy.*` 基于轻量代理。 | 需要更严谨画质分析时优先 `paper.*`。 |
| `paper.drift_clip_first_last` | `drift_clip.end` | 都在描述“从开头到结尾的语义变化”。 | `paper.*` 只比较明确的首 clip 和末 clip；`drift_clip.end` 来自“各 segment 相对首 segment”的定义。 | 要精确首尾漂移时看 `paper.*`；要更细的时序诊断时保留 `drift_clip.*`。 |
| `paper.repetition_clip_global` | `repetition_clip.mean_sim`, `repetition_clip.ratio`, `repetition_clip.max_sim` | 都在衡量语义重复。 | `paper.*` 是 all-pairs clip 平均，不依赖阈值；`repetition_clip.*` 有远距离 pair 过滤，也能反映阈值命中情况。 | 论文复现看 `paper.*`；排查循环/冻结细节时结合 `repetition_clip.*`。 |
| `paper.dinov2_cf.*` | `dinov2_lc.*` | 都用 DINOv2 衡量长时结构一致性。 | `paper.*` 按 Context Forcing 的窗口协议、相对首帧计算；`dinov2_lc.*` 是 segment mean feature 相对首 segment。 | 复现 Context Forcing 时优先 `paper.*`；工程上做稳定 proxy 时可继续使用 `dinov2_lc.*`。 |
| `paper.balance.mean` | 无 | 无 | `paper.balance.mean` 是新增的 summary 级组合指标，会先对 drift 和 repetition 做 min-max 归一化后再求和。 | 只适合同批模型一起比较，不适合单视频直接解释。 |

一个实用经验是：

- 如果你在写论文表格或做正式 benchmark，优先使用 `paper.*`。
- 如果你在排查失败原因，旧版 proxy 指标依然很有价值，尤其适合做补充诊断。

### 4.12 `efficiency.*`

这组字段来自可选的 runtime sidecar JSONL 文件。

可能出现的字段包括：

- `efficiency.runtime_sec`
- `efficiency.gpu_mem_peak_gb`
- `efficiency.latency_ms`
- `efficiency.kv_cache_gb`
- `efficiency.denoise_sec`
- `efficiency.decode_sec`
- `efficiency.io_sec`
- `efficiency.generation_fps = metadata.num_frames / runtime_sec`

解释方式：

- 这些字段反映的是成本，不是输出质量。
- 只有和质量、一致性指标结合起来看，才适合做效率与效果的权衡分析。

### 4.13 `vbench.*`

这组字段来自官方 VBench。工具包本身并不重实现这些指标，而是在 `custom_input` 模式下调用官方 VBench，保留原始产物，并把标准化后的 `vbench.*` 列合并回工具包报告。

当前已接入的维度包括：

- `vbench.aesthetic_quality`
- `vbench.imaging_quality`
- `vbench.subject_consistency`
- `vbench.background_consistency`
- `vbench.motion_smoothness`
- `vbench.dynamic_degree`

解释方式：

- `vbench.aesthetic_quality`：官方 VBench 对整体美学观感的判断。
- `vbench.imaging_quality`：官方 VBench 对画面质量的判断。若你需要官方 benchmark 质量分数，应优先看它，而不是 `quality_proxy.*`。
- `vbench.subject_consistency`：官方 VBench 对主体在时间上是否保持一致的判断。
- `vbench.background_consistency`：官方 VBench 对背景是否稳定、连贯的判断。
- `vbench.motion_smoothness`：官方 VBench 对运动是否平滑、自然的判断。
- `vbench.dynamic_degree`：官方 VBench 对运动强度或动态程度的判断。它可以帮助区分“稳定是因为生成正确”和“稳定只是因为几乎没动”。

报告行为：

- 只要官方结果可解析，`model_summary.csv` 就会合并模型级 `vbench.*` 列。
- 只有当官方输出中存在可稳定匹配回视频的 per-video 行时，`per_video_metrics.csv` 才会写入 `vbench.*`。
- 官方原始产物保留在 `vbench_raw/` 下。
- 工具包标准化后的汇总文件会写成 `vbench_merged_summary.csv/jsonl`，如有视频级结果则额外写出 `vbench_merged_per_video.csv/jsonl`。

## 5. 如何联合多个指标做专业分析

单个指标往往有歧义。真正有诊断价值的分析，通常来自多指标联合判断。

### 5.1 区分语义漂移和外观漂移

建议联合观察：

- `clip_f.*` 与 `drift_clip.*`
- `colorhist_lc.*` 与 `drift_colorhist.*`
- 可选地再对照 `dinov2_lc.*` 与 `drift_dinov2.*`
- 如果你想更贴近论文评测协议，还可以补看 `paper.dinov2_cf.*`

典型模式：

- `clip_f` 明显下降，但 `colorhist_lc` 仍很高：
  说明外观还像原来，但语义内容已经变了，常见于人物身份漂移、动作语义漂移、稳定背景下的人物失真。
- `colorhist_lc` 下降明显，但 `clip_f` 仍较高：
  说明大语义还在，但光照、色调、曝光或整体外观发生了较大变化。
- 两者一起恶化：
  往往意味着整体生成过程不稳定，出现了全局场景漂移。

### 5.2 区分“自洽但不符合 prompt”和“符合 prompt 但不稳定”

建议联合观察：

- `clip_t.*`
- `clip_f.*`

典型模式：

- `clip_f` 高、`clip_t` 低：
  视频内部比较自洽，但整体生成方向就是错的，属于“稳定地生成错内容”。
- `clip_t` 高、`clip_f` 低：
  每个局部片段可能都像 prompt，但长视频在人物、场景或状态延续上不稳定。
- 两者都高：
  往往说明模型同时具备较好的 prompt 跟随和长时一致性。
- 两者都低：
  模型在 prompt 对齐和长时稳定性两个维度都存在明显问题。

### 5.3 分析随时间退化的画质问题

建议联合观察：

- `paper.aesthetic_quality_drift_abs`
- `paper.imaging_quality_drift_abs`
- `paper.imaging_quality_drift_std`
- `quality_proxy.head`
- `quality_proxy.tail`
- `quality_proxy.delta_drop`
- `quality_proxy.segment_std`
- `drift_clip.end`

典型模式：

- `paper.imaging_quality_drift_abs` 或 `paper.aesthetic_quality_drift_abs` 很大：
  说明在官方质量打分器看来，视频结尾明显比开头差，或者整体风格质量已经发生明显退化。
- `paper.imaging_quality_drift_std` 很大：
  说明不只是首尾差，中间过程也存在明显的质量波动。
- `quality_proxy.delta_drop` 大且为正，同时 `drift_clip.end` 也高：
  说明后半段不仅画质变差，语义也开始明显跑偏。
- `quality_proxy.segment_std` 很大，但漂移并不特别严重：
  说明问题更像是局部闪烁、曝光波动、清晰度忽高忽低，而不是持续性语义崩坏。
- `quality_proxy` 看起来稳定，但 `clip_t` 很差：
  说明视频“看上去干净”，但语义上并没有按 prompt 来生成。

如果官方 VBench 可用，还可以进一步看：

- `vbench.imaging_quality` 高，但 `quality_proxy.mean` 偏低：
  说明轻量级 proxy 可能低估了视频质量，做 benchmark 比较时应更信任官方 VBench。
- `vbench.imaging_quality` 偏低，但 `quality_proxy.mean` 尚可：
  说明视频可能清晰度还行，但仍然存在伪影、细节不自然等轻量级 proxy 难以覆盖的问题。

### 5.4 检测冻结、循环与模式坍塌

建议联合观察：

- `paper.repetition_clip_global`
- 比较多个模型时再看 `paper.balance.mean`
- `repetition_clip.*`
- `repetition_colorhist.*`
- `clip_f.*`
- `quality_proxy.segment_std`

典型模式：

- 重复指标很高，同时 `clip_f` 也极高：
  很可能视频几乎冻结，只是在重复同一个状态。
- `paper.drift_clip_first_last` 很低，但 `paper.repetition_clip_global` 很高：
  说明模型看似“保持住了开头语义”，但很可能主要是因为它没有真正推进内容。
- `repetition_clip` 很高，而 `clip_f` 中低：
  可能不是完全冻结，而是在少数语义状态之间来回循环。
- `repetition_colorhist` 高，但 `repetition_clip` 没那么高：
  说明颜色和背景风格在重复，但语义内容仍有变化。

如果官方 VBench 可用，还建议对照：

- `vbench.motion_smoothness` 高，但 `vbench.dynamic_degree` 低：
  可能视频之所以“平滑”，只是因为过于静止。
- `vbench.dynamic_degree` 高，但 `vbench.motion_smoothness` 低：
  说明模型有足够运动量，但运动表现抖动、不稳定或不自然。

### 5.5 识别“开头好、结尾坏”的长时失败模式

建议重点看：

- `clip_f.drop`
- `clip_t.drop`
- `paper.aesthetic_quality_drift_abs`
- `paper.imaging_quality_drift_abs`
- `quality_proxy.delta_drop`
- `drift_clip.end`

这是长视频生成里非常常见的一种失败模式：

- 开头几段还不错，
- 首段质量也可以，
- 但到后半段，prompt 对齐下降、语义开始漂移、画质也逐步变差。

如果这三类 drop 都同时偏大，通常说明模型还不具备稳定的长时生成能力。

如果同时拿到了官方 VBench，还可以进一步结合：

- `vbench.subject_consistency`
- `vbench.background_consistency`

这样能帮助区分“后期主体语义漂移”和“后期整体画面质量退化”。例如，`vbench.subject_consistency` 很低而 `vbench.background_consistency` 还可以时，通常意味着主体先跑偏，而场景骨架仍相对稳定。

### 5.6 效率与质量的权衡分析

建议联合观察：

- `efficiency.runtime_sec`
- `efficiency.generation_fps`
- `efficiency.gpu_mem_peak_gb`
- 任意质量目标，如 `clip_f.mean`、`clip_t.mean`、`quality_proxy.mean`

实际使用建议：

- 优先找 Pareto frontier，而不是盲目追求单一指标最大。
- 一个更慢的模型，只有在一致性、文本对齐或稳定性上带来明显收益时，才值得额外成本。

## 6. 推荐分析流程

做模型比较时，一个实用流程是：

1. 先看 `model_summary.csv`，快速找到明显优劣势。
2. 联合看 `clip_f.mean.mean`、`clip_t.mean.mean`、`quality_proxy.mean.mean` 和各类 drift 指标。
3. 再下钻到 `per_video_metrics.csv`，检查不同 prompt 上的方差。
4. 按 `clip_t.drop` 最差、`drift_clip.end` 最高、`quality_proxy.delta_drop` 最大的样本排序。
5. 打开这些视频人工核验，确认到底是语义漂移、画质崩坏，还是重复/冻结问题。

这种“先 summary、再 per-video”的流程，能避免被单个 cherry-pick 样本误导。

## 7. 什么是 VBench，什么不是

本仓库提供了 VBench wrapper，但并未在工具包内部重实现官方 VBench 各维度。

因此：

- `quality_proxy.*` 不是 VBench `imaging_quality`。
- `clip_t.*` 不是官方 VBench 的文本对齐指标。
- `clip_f.*`、`colorhist_lc.*`、`drift_*`、`repetition_*`、`dinov2_*` 都是工具包内部代理指标。
- `paper.*` 是论文风格的派生指标。其中部分字段会复用官方 VBench 的 clip 级打分器，但它们本身仍不是原生整视频 `vbench.*` 字段。

如果你通过 wrapper 或 `longvideo-eval run --enable-vbench` 调用了官方 VBench，并在后处理阶段把结果合并进来，应当显式使用类似如下命名：

- `vbench.aesthetic_quality`
- `vbench.imaging_quality`
- `vbench.background_consistency`
- `vbench.subject_consistency`
- `vbench.motion_smoothness`
- `vbench.dynamic_degree`

这样可以在报告层面保持科学严谨，避免把工具包 proxy 与 benchmark score 混淆。

## 8. 实际使用中的注意事项

- 不要在分辨率、时长、prompt 集完全不同的条件下直接比较模型。
- 不要把任何单一代理指标当作绝对真理。
- 漂移低不一定代表视频好，也可能意味着视频几乎没动。
- prompt 对齐高也不一定代表动作顺序、数量关系和事件逻辑正确。
- 对重要结论，始终建议回看代表性视频，尤其是每类指标中最好和最差的样本。
