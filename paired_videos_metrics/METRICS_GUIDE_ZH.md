# 视频评测指标中文指南

本文档介绍 `LongVideoEvalKit` 中 `paired_videos_metrics` 子模块使用的四个常用成对视频指标。

推荐入口是：

```bash
longvideo-eval paired-run \
  --gt-dir /path/to/gt \
  --pred-dir /path/to/pred \
  --output-dir ./outputs/paired_eval \
  --device cuda
```

该子模块用于评测：

- GT 视频 vs 预测视频
- 模型 A 输出 vs 模型 B 输出
- 视频预测 / 重建 / 对齐生成任务

- `FVD`：Frechet Video Distance
- `SSIM`：Structural Similarity Index Measure
- `PSNR`：Peak Signal-to-Noise Ratio
- `LPIPS`：Learned Perceptual Image Patch Similarity

除了指标定义，本文还会说明：

- 每个指标到底在衡量什么
- 对应的核心计算公式
- 结果应该怎么分析
- 哪些任务适合看哪些指标
- 如何避免常见的评测误区
- 如何结合一个真实输出样例做解释

本文最后会使用下面这组张量级测试样例来做分析：

```python
NUMBER_OF_VIDEOS = 8
VIDEO_LENGTH = 30
CHANNEL = 3
SIZE = 64
videos1 = torch.zeros(NUMBER_OF_VIDEOS, VIDEO_LENGTH, CHANNEL, SIZE, SIZE, requires_grad=False)
videos2 = torch.ones(NUMBER_OF_VIDEOS, VIDEO_LENGTH, CHANNEL, SIZE, SIZE, requires_grad=False)
```

也就是：

- `videos1`：8 个全黑视频
- `videos2`：8 个全白视频
- 每个视频 30 帧
- 每帧大小 `64 x 64`
- 像素范围都在 `[0, 1]`

## 1. 先理解：这四个指标分别看什么

### 1.1 FVD

`FVD` 是视频版的 `FID`，用于衡量两组视频在高层语义特征空间中的分布差异。

它的特点是：

- 不直接逐像素比较
- 先把视频送进预训练的视频特征网络
- 再比较“真实视频集合”和“生成视频集合”的特征分布

它更适合回答这样的问题：

- 生成视频整体是否更像真实视频
- 动作、时序、语义是否更自然
- 模型生成的视频分布是否更接近真实分布

结论方向：

- `FVD` 越小越好

### 1.2 SSIM

`SSIM` 用来衡量两张图像在结构上的相似性。

它主要考虑：

- 亮度是否相近
- 对比度是否相近
- 局部结构是否相近

它比单纯的均方误差更符合人眼直觉，但本质上仍然是“配对图像比较”指标。

结论方向：

- `SSIM` 越大越好
- 理想值是 `1`

### 1.3 PSNR

`PSNR` 是基于像素误差的保真度指标。

它本质上来自 `MSE`，也就是均方误差。两张图逐像素越接近，`PSNR` 越高。

它适合回答：

- 重建结果是不是像素级接近真值
- 预测帧和 GT 帧是否精确对齐

结论方向：

- `PSNR` 越大越好

### 1.4 LPIPS

`LPIPS` 是感知相似度指标，它不是直接比较像素，而是比较深度网络提取到的感知特征。

它比 `PSNR/SSIM` 更接近人类视觉感受，尤其在以下情况更有用：

- 结果和 GT 没有逐像素严格对齐
- 图像看起来相似，但像素值差别不小
- 希望评估“看起来像不像”

结论方向：

- `LPIPS` 越小越好
- `0` 表示感知上几乎一致

## 2. 四个指标的核心公式

## 2.1 FVD 的公式

设：

- `f(x)` 表示视频 `x` 经过 I3D 提取后的特征
- 生成视频特征分布的均值和协方差为 `(mu_g, Sigma_g)`
- 真实视频特征分布的均值和协方差为 `(mu_r, Sigma_r)`

则：

```text
FVD = ||mu_g - mu_r||^2
    + Tr(Sigma_g + Sigma_r - 2 * (Sigma_g * Sigma_r)^(1/2))
```

其中：

- 第一项衡量两组视频特征均值差异
- 第二项衡量两组视频特征分布形状差异

直观理解：

- 如果生成视频和真实视频在语义特征空间里分布非常接近，`FVD` 就会小
- 如果两者分布差异大，`FVD` 就会大

## 2.2 SSIM 的公式

对两张图像 `x` 和 `y`：

```text
SSIM(x, y) =
((2 * mu_x * mu_y + C1) * (2 * sigma_xy + C2)) /
((mu_x^2 + mu_y^2 + C1) * (sigma_x^2 + sigma_y^2 + C2))
```

其中：

- `mu_x`, `mu_y` 是局部均值
- `sigma_x^2`, `sigma_y^2` 是局部方差
- `sigma_xy` 是局部协方差
- `C1`, `C2` 是稳定项

直观理解：

- `1` 附近：结构非常相似
- `0` 附近：结构差异很大

## 2.3 PSNR 的公式

先计算均方误差：

```text
MSE = mean((x - y)^2)
```

然后：

```text
PSNR = 20 * log10(MAX_I / sqrt(MSE))
```

在这个工具包中，像素范围是 `[0, 1]`，所以：

```text
MAX_I = 1
```

因此公式可以写成：

```text
PSNR = 20 * log10(1 / sqrt(MSE))
```

直观理解：

- 图像完全相同，`MSE` 很小，`PSNR` 就高
- 图像误差越大，`PSNR` 越低

## 2.4 LPIPS 的公式

LPIPS 的思想是比较深度网络多个层的特征差异，可概括写成：

```text
LPIPS(x, y) = sum_l w_l * mean((normalize(phi_l(x)) - normalize(phi_l(y)))^2)
```

其中：

- `phi_l(x)` 是第 `l` 层提取的特征
- `normalize` 表示特征归一化
- `w_l` 是每层的学习权重

直观理解：

- 越接近 `0`，说明感知上越像
- 值越大，说明视觉差异越明显

## 3. paired-run 的输入、配对和汇总方式

### 3.1 CLI 输入形式

在 `paired-run` 中，输入不是直接传张量，而是两个目录：

- `--gt-dir`
- `--pred-dir`

工具会自动读取目录中的视频文件并尝试配对。

### 3.2 配对规则

配对顺序固定为：

1. 完整文件名精确匹配
2. 去掉扩展名后的 stem 精确匹配
3. 去掉扩展名后的 stem 从后向前后缀匹配

配对过程中会生成：

- `pairing_report.json`
- `pairing_report.csv`

其中会统一记录：

- `non_exact_match_pairs`
- `unmatched_gt_videos`
- `unmatched_pred_videos`

### 3.3 时长对齐规则

如果配对成功的两个视频时长不一致：

- 工具会取较短时长作为对齐长度
- 将长视频从开始位置截取到同样时长
- 再做逐帧和视频级指标计算

相关字段包括：

- `gt_duration_sec`
- `pred_duration_sec`
- `aligned_duration_sec`
- `duration_truncated`

### 3.4 底层张量格式

底层函数仍然采用如下张量格式：

```text
[B, T, C, H, W]
```

也就是：

- `B`：视频个数
- `T`：帧数
- `C`：通道数
- `H, W`：高和宽

### 3.5 对 SSIM / PSNR / LPIPS

这三个指标的计算逻辑是：

1. 对每个视频的每一帧分别计算
2. 再对帧求平均
3. 再对视频求平均

如果 `only_final=True`，最后只返回一个整体均值。

如果 `only_final=False`，则会返回一个随时间累积的结果序列。

在 `paired-run` 最终输出中，对每条视频会产生：

- `psnr_mean`
- `ssim_mean`
- `lpips_mean`

在 `summary.json` 中，对整个数据集会按帧数加权汇总。

### 3.6 对 FVD

`FVD` 不是逐帧算后平均，而是：

1. 先把整段视频抽成 I3D 特征
2. 再把两组视频的特征分布做 Frechet 距离

在 `paired-run` 输出中：

- 每条视频有一个 `fvd_pair`
- 整个数据集还有一个 `fvd_dataset`

两者的解释不同：

- `fvd_pair`：单个 GT / Pred 视频对之间的视频级差异
- `fvd_dataset`：整个 GT 集合与整个 Pred 集合之间的分布级差异

这两个值可能给出不同倾向，实际分析时应该一起看。

## 4. 输出文件怎么看

典型输出目录包含：

```text
output_dir/
  config.json
  pairing_report.json
  pairing_report.csv
  per_video_metrics.jsonl
  per_video_metrics.csv
  summary.json
  final_results.json
```

建议读取顺序：

1. `config.json`
   确认设备、匹配视频数、是否跳过某些指标
2. `pairing_report.json`
   确认是否存在大量非精确匹配或未匹配视频
3. `summary.json`
   看整体结果
4. `per_video_metrics.csv`
   排查具体好坏样本
5. `final_results.json`
   需要程序化处理时使用

## 5. 什么时候该看哪个指标

### 5.1 如果任务是“配对评测”

比如：

- 视频预测
- 视频重建
- 有明确 GT 的条件生成

这时更适合重点看：

- `PSNR`
- `SSIM`
- `LPIPS`

因为这些指标默认假设：

- 生成结果和真值是一一对应的
- 每一帧都应该对得上

### 5.2 如果任务是“开放生成”或“分布评测”

比如：

- 文本生成视频
- 无唯一真值的视频生成
- 更关注 realism 和 temporal quality 的任务

更适合重点看：

- `FVD`

因为开放生成任务中，一个 prompt 可能存在多个合理答案。此时逐像素比较往往不公平。

## 6. 怎么正确分析评测结果

## 6.1 先看指标方向

记住这四个指标的方向：

- `FVD`：越小越好
- `LPIPS`：越小越好
- `PSNR`：越大越好
- `SSIM`：越大越好

## 6.2 不要只看单一指标

单个指标往往不够。

比较常见的组合解读方式：

- `PSNR` 高，`SSIM` 高，`LPIPS` 低：通常说明重建质量较好
- `FVD` 低：通常说明整体视频分布、语义和时序更接近真实视频
- `PSNR` 高但 `FVD` 不好：说明逐帧接近，但视频整体动态或语义分布未必自然
- `LPIPS` 好但 `PSNR` 一般：说明感知上像，但像素级不完全对齐

## 6.3 不要随意跨设置比较绝对值

很多人会直接拿不同实验的绝对数值横比，这很容易误判。

比较前必须确认这些条件一致：

- 视频数量是否一致
- 帧数是否一致
- 分辨率是否一致
- 像素范围是否一致
- 是否是配对评测
- `FVD` 用的是哪种 backend
- 是否都用 `only_final=True`

只有评测 protocol 一致，结果才有可比性。

## 6.4 更适合做“相对比较”

尤其是 `FVD` 和 `LPIPS`，更适合这样理解：

- 同一个数据集、同一个流程下，模型 A 比模型 B 更低，则 A 更好

而不是：

- “`FVD=120` 就一定很好”
- “`LPIPS=0.2` 就一定够用了”

这些指标的绝对值很依赖任务和数据分布。

## 7. 用刚才的输出做实例分析

刚才测试得到的结果是：

```json
{
  "fvd": {
    "value": [151.39240551901463]
  },
  "ssim": {
    "value": [9.99900009999073e-05],
    "value_std": [2.710505431213761e-20]
  },
  "psnr": {
    "value": [0.0],
    "value_std": [0.0]
  },
  "lpips": {
    "value": [0.8140428066253662],
    "value_std": [0.0]
  }
}
```

测试输入是：

- 一组全黑视频
- 一组全白视频

这是一组非常极端、非常容易人工判断的样例，因此非常适合做 sanity check。

### 7.1 为什么 PSNR = 0 完全合理

因为：

```text
videos1 中每个像素 = 0
videos2 中每个像素 = 1
```

所以：

```text
MSE = mean((0 - 1)^2) = 1
```

代入公式：

```text
PSNR = 20 * log10(1 / sqrt(1)) = 0
```

所以 `PSNR = 0.0` 是严格正确的，不是异常值。

这说明：

- 两组视频在像素级完全不接近

### 7.2 为什么 SSIM 接近 0

`SSIM = 9.999e-05`，非常接近 `0`。

这是合理的，因为：

- 一边是纯黑
- 一边是纯白
- 几乎没有共享结构信息
- 亮度和对比关系也完全不同

之所以不是精确的 `0`，是因为 SSIM 公式里有数值稳定项 `C1` 和 `C2`。

所以应该把这个结果理解为：

- 结构相似性几乎不存在

### 7.3 为什么 LPIPS 大约是 0.814

`LPIPS = 0.8140428066253662`

这个值说明：

- 从感知特征上看，两组图像差异很大

这也符合直觉：

- 全黑图像和全白图像在人眼看来当然差别极大

如果两组图像非常像，LPIPS 应该更接近 `0`。这里在 `0.81` 左右，说明感知差距明显。

### 7.4 为什么 FVD 大约是 151.39

`FVD = 151.3924`

这个结果也合理，而且和该仓库 README 中给出的 sanity check 数值非常接近。

你可以这样理解：

- 两组视频在视频语义特征空间中相差很大
- 一组是全黑视频分布
- 一组是全白视频分布
- 即使它们都“没有运动”，它们在 I3D 特征空间中的分布仍然明显不同

这里不要把 `151` 理解为某种百分比。

`FVD` 更适合做相对比较：

- 如果模型 A 的 `FVD` 比模型 B 低，说明 A 生成的视频分布更接近参考分布

对这个极端测试样例来说，`151` 左右就是一个合理的 baseline 数值。

### 7.5 为什么标准差几乎是 0

这里：

- `PSNR` 的标准差是 `0`
- `LPIPS` 的标准差是 `0`
- `SSIM` 的标准差接近 `0`

原因很简单：

- 每一对视频都是同样的“全黑 vs 全白”
- 每一帧也是同样的“全黑 vs 全白”

所以所有样本的指标值几乎完全相同，不会有波动。

## 8. 真实实验中应该怎么读结果

## 8.1 如果你做的是视频预测或重建

优先看：

- `PSNR`
- `SSIM`
- `LPIPS`

建议解读方式：

- `PSNR` 看像素级保真度
- `SSIM` 看结构恢复是否稳定
- `LPIPS` 看视觉观感是否更接近真值

一个常见现象是：

- 有些模型 `PSNR` 很高，但画面发糊
- 有些模型 `LPIPS` 更好，看起来更自然，但 `PSNR` 未必最高

所以最好三者一起看。

## 8.2 如果你做的是开放式视频生成

优先看：

- `FVD`

例如：

- text-to-video
- image-to-video 的开放生成场景
- 没有唯一标准答案的视频任务

因为在这类任务中：

- `PSNR/SSIM` 往往对“语义正确但像素不同”的结果不友好

这时 `FVD` 更适合衡量整体质量。

## 8.3 如果几个指标相互矛盾

这非常常见，不一定说明评测错了。

比如：

- `PSNR` 高，`LPIPS` 不好：像素平均误差不大，但视觉细节不自然
- `LPIPS` 好，`PSNR` 一般：视觉上很像，但没有做到逐像素对齐
- 帧级指标不错，`FVD` 一般：单帧好，但视频整体动态和分布还不够自然

这类情况下，应该回到任务目标：

- 你更关心精确重建，还是更关心真实感与时序自然性

## 9. 做模型横向比较时的建议

如果你要比较多个模型，建议统一以下设置：

1. 使用相同测试集
2. 使用相同视频数量
3. 使用相同帧数
4. 使用相同分辨率
5. 使用相同像素归一化方式
6. 使用相同 FVD backend
7. 使用相同 `only_final` 设置

建议最终汇报时至少写清楚：

- 数据集名称
- 测试样本数
- 每个视频的帧数
- 分辨率
- 是否 paired
- FVD backend 是 `styleganv` 还是 `videogpt`

## 10. 常见坑

### 10.1 像素范围错了

这个工具包要求输入像素范围是：

```text
[0, 1]
```

如果你传的是 `[0, 255]` 或 `[-1, 1]`，结果会明显失真。

### 10.2 输入 shape 错了

仓库要求：

```text
[B, T, C, H, W]
```

不是：

- `[B, C, T, H, W]`
- `[T, C, H, W]`

### 10.3 FVD 的视频长度不够

这个仓库要求：

```text
T >= 10
```

否则 FVD 无法正确计算。

### 10.4 开放生成任务滥用 PSNR/SSIM

如果一个 prompt 对应多个合理答案，生成视频和 GT 不逐像素一致并不代表结果差。

这时过度依赖 `PSNR/SSIM` 容易误判模型能力。

### 10.5 不同评测设置混着比

### 10.6 忽略配对报告

如果：

- 非精确匹配数量很多
- 未匹配 GT 或 Pred 数量很多

那你在解释 `summary.json` 之前，必须先确认：

- 当前参与评测的样本集合是否真的符合你的预期

否则很容易把“配对问题”误当成“模型问题”。

### 10.7 忽略时长截断

如果 `duration_truncated = true` 的样本很多，说明：

- 两边视频时长经常不一致

这时要小心解释 `FVD` 和帧级指标，因为：

- 你评测的是“对齐后的公共前缀”
- 不是整段原始视频的完整对比

如果你：

- 改了分辨率
- 改了帧数
- 换了 FVD backend
- 换了测试集

那结果就不能直接横比。

## 11. 一个简短总结

可以用一句话记住这四个指标：

- `PSNR`：像素误差有多小
- `SSIM`：结构有多像
- `LPIPS`：看起来有多像
- `FVD`：整组视频分布有多像真实视频

对刚才的 `全黑 vs 全白` 测试样例：

- `PSNR = 0`：像素级完全不相似
- `SSIM ~ 0`：结构几乎完全不相似
- `LPIPS ~ 0.814`：感知差异很大
- `FVD ~ 151.39`：视频分布差异明显

因此，这组输出是自洽的，也说明这套评测流程工作正常。
